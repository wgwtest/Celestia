# Celestia MVC Step12-18 Real Scene Projection Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Do not start implementation from this document before checking the current `master` status, reading `CODEX_START_HERE.md`, and creating the concrete Step-level worktree or branch.

**Goal:** 把 Celestia 当前已经完成的本机多进程 MVC runtime 骨架，推进到可以由独立 Model host 输出真实 Celestia 场景、由独立 View3D host 渲染真实场景、由 Controller 完成真实交互闭环的阶段。

**Architecture:** 后续推进顺序是 `契约 -> 真实 Model -> 真实 scene.frame -> View3D 渲染真实场景 -> Controller 交互 -> 资源/data-plane -> 能力保真硬化`。Model 核心语义不为 View 重写；View3D 以历史 View3D / Renderer 为参考实现和可拆零件来源，但新跨进程 View3D host 只消费 `scene.frame`、`ResourceRef`、View 配置和输入事件，不持有 `Simulation*`、`Universe*` 或 `CelestiaCore*`。

**Tech Stack:** C++17, CMake, doctest, PowerShell, `tools/mvc` dependency scans, `tools/regression` compatibility harness, Celestia `src/celengine`, Celestia `src/celruntime`, SDL/OpenGL local runtime.

## 2. 本计划的核心判断

后续问题的根部不在 View 画得少，而在 View 还没有拿到真实 Celestia 场景数据。

因此后续主线必须先解决：

```text
真实 Universe / Simulation / Observer / Selection
  -> 只读 scene projection
  -> scene.frame vNext
  -> ResourceRef
  -> View3D host renderable scene
```

不要把下一阶段误判为“大规模扩展截图场景”或“先美化当前 placeholder View”。在 View3D 尚未消费真实 `scene.frame` 之前，大规模截图矩阵只能证明简化 View 的占位行为，不足以证明 Celestia 原能力迁移。

## 4. Model Lock Policy

Model 锁死分三层，不是一个瞬间完成的动作。

| 锁级别 | 生效阶段 | 含义 |
| --- | --- | --- |
| 结构锁 | 已从 Step3 / Step4 生效 | `celestia_model` 的源码归属、CMake target、依赖方向已经固定；Model 不反向依赖 Controller / View / Renderer / SDL / OpenGL |
| 核心语义锁 | 从 Step12 开始执行，到 Step14 完成后基本成立 | 天体、星表、轨道、时间线、坐标、参考系等核心语义不为 View 改写；允许调整 `scene.frame`、`SceneExtractor`、Model facade 和只读投影 |
| Model 层整体冻结 | Step17 完成后进入，Step18 前确认 | `ModelService`、`SceneExtractor`、scene/resource 输出契约、Model command 行为整体冻结；后续只允许带回归证据的 bug fix |

```text
让 View3D host 直接读取 Model 进程内对象。
让 Controller 绕过 protocol 直接操作 Model 内部对象。
```

## 5. 新 View3D 实现原则

新的跨进程 View3D host 不是把现有 in-process View3D 整体搬进另一个进程。

正确关系是：

```text
历史 View3D / Renderer
  -> 参考实现
  -> 可拆算法来源
  -> 可拆 shader/resource/render utility 来源

新 celestia-view3d-host
  -> 独立进程
  -> 消费 scene.frame / ResourceRef / View config
  -> 产生画面和 view.input
  -> 不持有 Model 内部对象
```

允许复用或迁移：

```text
数学计算
投影逻辑
可独立的渲染算法
shader
纹理和 mesh 加载思路
celrender 中可以保持纯 View 侧语义的底层能力
```

禁止作为新 View3D host 依赖：

```text
CelestiaCore*
Simulation*
Universe*
Selection*
旧 AppWindow 事件循环
直接访问 Model 内存对象的 Renderer 入口
```
