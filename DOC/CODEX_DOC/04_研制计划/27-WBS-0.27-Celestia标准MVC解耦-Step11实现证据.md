# WBS-0.27 Celestia 标准 MVC 解耦 Step11 实现证据

日期：2026-06-25

## 1. 阶段目标

Step11 的目标不是继续扩大功能面，而是对 Step8 至 Step10 的运行时成果做工程收口：

- 移除早期用于验证多进程形态的兼容回退路径，避免后续误把临时方案当作正式协议。
- 固化 M / C / V runtime 装配、View 插件切换、跨进程 OpenGL3D View 的验收样例。
- 增加边界扫描脚本，使后续目录、target、接口调整可以快速发现反向依赖或历史兼容残留。
- 补齐最终架构说明，明确当前仓库内已经达到的能力，以及尚未承诺的外部化能力。

## 2. 代码收口内容

### 2.1 移除历史兼容回退

已从生产路径中移除以下 Step6 早期文件脚本式运行通道：

- `stdio-files` host transport。
- `ProcessSupervisor::runServeSmoke()`。
- SDL 启动侧的 `sdl-step6-serve` 分支。

当前正式 runtime transport 入口统一为：

- `stdio-pipe`
- `local-socket`

这意味着后续 M / C / V 进程间协议不再以文件轮询作为隐式兼容层，所有运行时装配必须通过正式 transport 与 `RuntimeSession` 完成。

### 2.2 固化运行时样例

新增两个可长期保留的 runtime config 样例：

- `DOC/CODEX_DOC/examples/runtime-3d-stdio.yaml`
- `DOC/CODEX_DOC/examples/runtime-switch-3d-to-2d-local-socket.yaml`

它们覆盖：

- OpenGL3D View 跨进程启动。
- 2D / 3D View 在同一 runtime assembly 中的切换。
- `stdio-pipe` 与 `local-socket` 两条正式 transport 路径。

### 2.3 增加自动扫描

新增扫描脚本：

- `tools/mvc/scan_mvc_dependencies.ps1`
- `tools/mvc/scan_cmake_targets.ps1`

扫描范围包括：

- MVC protocol 头文件纯净度。
- 生产源码中是否重新出现已移除的 Step6 文件脚本式回退。
- `celruntime` / `celengine` / `celrender` 相关 CMake target 边界。
- 旧 forwarding header 与目录重组遗留风险。

## 3. 文档收口内容

新增最终架构说明：

- `DOC/CODEX_DOC/02_设计说明/02-07-Celestia标准MVC最终架构说明.md`

更新协议说明：

- `DOC/CODEX_DOC/02_设计说明/02-06-Celestia-MVC-Runtime-Protocol-v1.md`

文档口径统一为：

- 当前仓库内已经形成 local multi-process MVC runtime。
- Model / Controller / View 具备独立 host 进程运行与 runtime assembly 装配能力。
- View 已经通过 registry / plugin manifest / runtime config 进入可替换路径。
- OpenGL3D View 已经可以作为跨进程 View host 被 supervisor 启动和停止。
- 尚未把该能力扩大为跨机器 TCP 服务、第三方插件生态、完整共享内存资产流或完整历史 OpenGL 功能等价。

## 5. 当前能力边界

可以确认的当前能力：

- M / C / V 已经可以在本机由 supervisor 分别启动为独立 host 进程。
- Controller 可以通过 runtime assembly 驱动 Model 与 View。
- Debug2D View 与 OpenGL3D View 可以通过统一 runtime config 路径装配。
- View registry / plugin manifest 已经提供 View 替换与切换入口。
- 早期文件脚本式兼容回退已经从生产路径移除。

不能扩大表述为：

- 已经形成跨机器长期运行三服务架构。
- 任意第三方 View 已经无需适配即可热插拔。
- 所有历史 OpenGL 渲染能力已经在跨进程 View 中完全等价。
- 纹理、mesh、星表等重型资产已经全部进入共享内存或零拷贝数据面。

## 6. 结论

Step11 将 Step8 至 Step10 的实验性 runtime 成果收口为当前仓库内可验证、可演示、可继续扩展的标准 MVC runtime 基线。

下一步如果继续推进，应从“仓库内本机 runtime”转向“长期运行服务化 runtime”，重点工作不再是拆目录，而是完善生命周期、错误恢复、资源数据面、协议版本协商、性能基准和真实 View 插件加载策略。
