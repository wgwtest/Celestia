# Celestia MVC Step10 Unified Runtime Assembly and High-Performance IPC Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将 Step7 的 stdio-pipe 验证级主路径升级为可替换 transport、统一 runtime 装配、高频数据面可扩展的长期运行框架。

**Architecture:** Step10 把协议语义、传输实现和 runtime 装配三者分开。`RuntimeEnvelope` 和 scene protocol 保持为协议层；stdio-pipe、local socket、tcp socket、shared memory 等变为 transport/data-plane 插件；RuntimeSession 只依赖 transport 抽象和 assembly config，不关心底层 IPC 类型。高频渲染数据区分控制面和数据面，避免把所有 scene payload 永久塞进同步 stdio 管道。

**Tech Stack:** C++17, CMake, doctest, existing RuntimeEnvelope/FramedMessage, Windows named pipe or local socket first implementation, optional loopback TCP, future shared memory data plane.

---

## 1. 阶段定位

Step10 建立在：

```text
Step7: M / C / V 三 host 长期 stdio-pipe IPC 和进程监管。
Step8: 3D View 跨进程 scene protocol。
Step9: View plugin ABI 和有序切换。
```

Step10 解决的是：

```text
同一套 runtime 配置如何装配 Model / Controller / View / transport / resource root。
stdio-pipe、local socket、tcp socket 等 transport 如何替换。
控制面和高频数据面如何分离。
RuntimeSession 如何从固定 host 启动逻辑变成统一 assembly executor。
长时间运行的 supervisor、heartbeat、restart、trace 如何形成稳定机制。
```

Step10 不解决：

```text
彻底移除所有兼容头。
任意公网分布式部署。
完整第三方插件生态治理。
渲染算法质量重写。
```

## 2. 目标能力

Step10 完成后可以说：

```text
2D / 3D runtime 可以由统一 assembly config 装配。
transport 可以替换，且不影响 Model / Controller / View 业务代码。
至少 stdio-pipe 和 local socket 两种本机 transport 可运行。
runtime 明确区分 lifecycle/control message 与 high-frequency scene data。
long-running supervisor 具备 heartbeat、timeout、restart policy、shutdown policy 和 trace。
```

Step10 完成后仍不能说：

```text
所有历史兼容层已经移除。
所有跨机器部署问题已经解决。
shared memory 已经覆盖所有渲染数据。
View ABI 永久冻结。
```

## 3. Runtime Assembly Config

建议配置结构：

```yaml
session:
  id: celestia-runtime
  durationMs: 0
  heartbeatMs: 1000
  shutdownTimeoutMs: 3000

  contentRoot: build-mvc-sdl-rel/run-full

  control:
    kind: stdio-pipe
  data:
    kind: framed-envelope

  host: celestia-model-host
  args: ["--serve"]

  host: celestia-controller-host
  args: ["--serve"]

  id: celestia.view3d.opengl
  pluginDir: build-mvc-sdl-rel/plugins
  host: celestia-view3d-host
  args: ["--serve"]
```

配置约束：

```text
协议版本必须显式。
resource root 必须显式。
transport kind 必须显式。
view id 必须通过 plugin registry 验证。
host path 必须可以解析为实际可执行文件。
```

## 4. Transport 抽象

```cpp
class RuntimeTransport
{
public:
    virtual ~RuntimeTransport() = default;
    virtual bool open() = 0;
    virtual bool send(const protocol::RuntimeEnvelope&) = 0;
    virtual std::optional<protocol::RuntimeEnvelope> receive(std::chrono::milliseconds timeout) = 0;
    virtual void close() = 0;
    virtual RuntimeTransportStats stats() const = 0;
};
```

第一轮 transport：

```text
stdio-pipe:
  复用 Step7 HostProcess。

local socket:
  本机 socket 或 Windows named pipe 二选一，按现有依赖和实现成本决定。

tcp socket:
  可作为后续扩展，第一轮只需要接口不阻塞实现。
```

## 5. 控制面和数据面


数据面：

```text

```

Step10 第一轮可以仍然让 `scene.frame` 通过 framed transport，但必须建立 data-plane abstraction：

```text
DataPlaneRef
  kind
  id
  byteLength
  contentHash
  transportHint

DataPlaneChannel
  publish()
  acquire()
  release()
```

这样 Step10 后续可以把大块数据迁移到 shared memory，而不改变 Model/View 协议语义。

## 9. 风险和缓解

| 风险 | 表现 | 缓解 |
| --- | --- | --- |
| transport 抽象污染协议 | 每种 IPC 都改消息语义 | RuntimeEnvelope/scene protocol 不依赖 transport kind |
| 过早做复杂 shared memory | Step10 无法收口 | 先建 DataPlane contract，再做最小 Windows shared memory |
| local socket 和 tcp socket 概念混淆 | 配置和验收口径不清 | local-socket 表示本机 IPC，tcp-socket 表示可远程连接 |
| restart policy 误伤调试 | host 崩溃被自动重启掩盖 | trace 必须记录 crash/restart，默认策略保守 |
| config 取代 CLI 导致兼容破坏 | 旧启动命令失效 | CLI 映射到 RuntimeAssemblyConfig，保留旧参数 |
