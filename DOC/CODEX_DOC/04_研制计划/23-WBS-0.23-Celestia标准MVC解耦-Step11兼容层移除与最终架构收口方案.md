# Celestia MVC Step11 Compatibility Layer Removal and Final Architecture Closure Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在 Step8-Step10 的运行能力完成后，移除迁移期兼容头、临时 adapter、旧 include 路径和不合理 CMake 依赖，完成标准 MVC 解耦的最终架构收口。

**Architecture:** Step11 不再引入新的大能力，而是做边界收紧、依赖清理、测试闭环和文档固化。Model / Controller / View / Runtime / Protocol 只通过正式 public API 通信；历史兼容层必须有清单、有替代路径、有删除提交、有回归验证。最终交付要让代码结构、CMake target、文档图、运行命令和验收测试一致。

**Tech Stack:** C++17, CMake, doctest, `rg`, dependency scan scripts, existing MVC runtime/process/plugin/transport modules.

---

## 1. 阶段定位

Step11 只能在以下前置能力已经完成后启动：

```text
Step8: 3D View 已可跨进程运行。
Step9: View plugin ABI 和有序切换已完成。
Step10: runtime assembly 和 transport abstraction 已完成。
```

Step11 解决的是：

```text
迁移期留下的兼容头是否可以删除。
M / C / V 是否还存在反向 include 或跨层直接依赖。
CMake target 是否还存在过宽链接。
文档里的依赖图、运行图、实际代码是否一致。
最终验收命令是否覆盖 in-process、multi-process 2D、multi-process 3D、View 切换、transport 切换和长时间运行。
```

Step11 不解决：

```text
新的渲染功能。
新的 View 插件。
新的 IPC 类型。
新的 UI 交互能力。
```

## 2. 最终架构口径

Step11 完成后才可以说：

```text
Celestia 标准 MVC 解耦已经完成到当前定义的最终目标。
M / C / V 可以作为长期独立进程运行。
View 可以通过插件机制装配和替换。
2D / 3D View 走统一 runtime assembly。
transport 可替换且不污染业务协议。
历史兼容头和迁移 shim 已被清理到文档允许范围。
```

如果以下任一项不成立，不能宣布最终完成：

```text
Model 仍直接 include View 具体实现。
View 仍直接 include Model 内部对象。
Controller 仍直接操作 Renderer 或 Simulation 内部对象。
RuntimeSession 仍写死某个具体 View。
旧兼容头仍是主路径必须依赖。
CMake target 仍通过一个大 target 间接链接全部模块。
```

## 4. 目标依赖边界

允许依赖：

```text
Model -> Protocol
Model -> Runtime host loop
Controller -> Protocol
Controller -> Runtime host loop
View -> Protocol
View -> Runtime host loop
Runtime -> Protocol
Runtime -> Process/Transport/DataPlane
Launcher -> Runtime assembly
Plugin loader -> View plugin ABI
```

禁止依赖：

```text
Model -> View concrete implementation
Model -> Controller concrete implementation
Controller -> Model concrete implementation internals
Controller -> View concrete implementation internals
View -> Model concrete implementation internals
View -> Controller concrete implementation internals
Protocol -> Renderer/OpenGL/SDL
Protocol -> Process/Transport concrete implementation
```

最终 target 方向：

```text
celestia_protocol
  no dependency on model/controller/view/runtime

celestia_model
  depends on celestia_protocol

celestia_controller
  depends on celestia_protocol

celestia_view_api
  depends on celestia_protocol

celestia_runtime
  depends on celestia_protocol, process, transport, dataplane

celestia_view2d_debug_plugin
  depends on celestia_view_api, celestia_protocol

celestia_view3d_opengl_plugin
  depends on celestia_view_api, celestia_protocol, renderer/OpenGL/SDL

celestia-sdl
  depends on celestia_runtime and launcher glue
```

## 9. 风险和缓解

| 风险 | 表现 | 缓解 |
| --- | --- | --- |
| 过早删除兼容层 | 外部入口或测试突然断裂 | 先建 inventory，逐组删除，逐组构建和提交 |
| 清理变成大重构 | 难以定位回归 | 每组兼容层独立提交，避免混入新功能 |
| CMake target 收紧导致隐式依赖暴露 | 构建失败很多 | 按 target 方向逐步处理，禁止重新用大 target 掩盖问题 |
| 文档和代码再次不一致 | 用户无法判断真实状态 | 最终文档必须引用真实命令、输出、截图和扫描结果 |
| Typora 图打不开 | 文档不可用 | Mermaid 使用保守语法，提交前手工打开或渲染检查 |
