# Celestia MVC Step9 View Plugin ABI and Hot-Swap Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将 2D / 3D View 从编译期固定绑定推进为 runtime 可发现、可加载、可查询能力、可切换的 View 插件体系。

**Architecture:** Step9 建立 View plugin ABI、manifest、loader、capability model 和 lifecycle contract。Runtime 只依赖稳定 ViewProvider 接口，不直接依赖具体 2D / 3D View 实现。热插拔第一轮限定为同一机器、同一进程族内的有序停启和重新绑定，不承诺任意时刻无损热替换。

**Tech Stack:** C++17, CMake, platform dynamic library loading (`LoadLibraryW` first), JSON or existing config parser, Step7/Step8 runtime protocol, doctest.

---

## 1. 阶段定位

Step8 完成后，系统应已经具备：

```text
2D debug view 可以跨进程运行。
3D OpenGL view 可以作为独立 View3D host 运行。
Model 和 View 之间通过 scene protocol 交互。
```

Step9 解决的是：

```text
View 如何从固定可执行入口变成 runtime 可装配单元。
2D / 3D View 如何用同一套生命周期和能力描述。
runtime 如何根据配置选择、加载、停止、切换 View。
View 实现如何在不修改 Model / Controller 的情况下替换。
```

Step9 不解决：

```text
shared memory 高性能数据面。
跨机器远程 View。
完整资源包市场或第三方插件生态。
无中断实时热替换。
移除全部兼容头。
```

## 2. 目标能力

Step9 完成后可以说：

```text
View 已经具备插件化装配基础。
runtime 可以通过 manifest 发现 View。
runtime 可以查询 View capability。
2D debug view 和 3D OpenGL view 走同一套 ViewProvider 生命周期。
在有序停止当前 View 后，可以切换到另一个 View。
Model 和 Controller 不需要知道具体 View 类型。
```

Step9 完成后仍不能说：

```text
任意第三方 View ABI 永久稳定。
View 可以任意时刻无损热替换。
所有 IPC transport 和数据面已经统一优化。
所有历史兼容头已经移除。
```

## 3. View 插件模型

插件由两部分组成：

```text
manifest:
  描述插件 id、类型、入口、capability、协议版本、资源需求。

binary/provider:
  提供稳定 C ABI 入口，返回 ViewProvider。
```

建议 manifest 字段：

```text
id
name
kind
version
protocolVersion
entry
processMode
capabilities
supportedTransports
resourceRequirements
defaultOptions
```

示例：

```json
{
  "id": "celestia.view3d.opengl",
  "name": "Celestia OpenGL 3D View",
  "kind": "view3d",
  "version": "0.1.0",
  "protocolVersion": 1,
  "entry": "celestia-view3d-plugin.dll",
  "processMode": "host-process",
  "capabilities": ["scene.frame", "view.input", "opengl", "sdl-window"],
  "supportedTransports": ["stdio-pipe"],
  "resourceRequirements": ["content-root"],
  "defaultOptions": {
    "showStars": true,
    "showOrbits": true
  }
}
```

## 4. ViewProvider ABI

稳定 C ABI 第一轮只暴露少量函数：

```cpp
extern "C" CELESTIA_VIEW_PLUGIN_API int celestia_view_plugin_abi_version();
extern "C" CELESTIA_VIEW_PLUGIN_API const char* celestia_view_plugin_id();
extern "C" CELESTIA_VIEW_PLUGIN_API CelestiaViewProvider* celestia_create_view_provider();
extern "C" CELESTIA_VIEW_PLUGIN_API void celestia_destroy_view_provider(CelestiaViewProvider*);
```

Provider 生命周期：

```text
load
queryCapabilities
configure
start
handleEnvelope
pollEnvelope
stop
shutdown
unload
```

约束：

```text
ABI 层只使用 C-compatible 类型或 opaque handle。
跨 ABI 不传 STL 容器。
跨 ABI 不传 C++ exception。
跨 ABI 不传 OpenGL object ownership。
跨 ABI 不传 Model / Controller 内部指针。
```

## 8. 风险和缓解

| 风险 | 表现 | 缓解 |
| --- | --- | --- |
| ABI 过早复杂化 | 插件系统拖慢 Step9 | 第一轮只暴露最小 C ABI 和 opaque handle |
| 热插拔定义过宽 | 用户误以为任意时刻无损替换 | 文档明确第一轮是有序 stop/load/start |
| 插件崩溃拖垮 runtime | View 插件异常导致 supervisor 卡死 | loader 和 lifecycle 必须返回错误，host 仍由 Step7 supervisor 监管 |
| STL 跨 DLL 边界 | ABI 不稳定或崩溃 | ABI 层禁止 STL/exception，内部 C++ wrapper 自行转换 |
| 2D/3D 两套装配分裂 | 后续无法统一 runtime | registry 按 view id + capability 统一选择 |
